# Retrospective: req-009

## Summary
req-009 was a deliberate smoke test following the much larger req-008 self-audit run. The user-facing work — inserting a 3-line "Why couch potato?" section into README.md — landed cleanly: Architect reviewed wording, Coder made the edit, Team Lead verified. No work-level friction, no escalations, no stagnation. However, Team Lead observed three process-level friction points during Initialization and Execute that are worth capturing. Two are documentation-correctness bugs in protocol.md and SKILL-body.md where the current text makes claims that are demonstrably false against the actual harness behavior; the third is a minor harness quirk with TaskCreate id ordering that is not actionable for couch-potato.

## Friction Analysis

### Friction 1 — TeamCreate `team_name` is overridden by the harness
- **What happened**: Team Lead called `TeamCreate team_name: "req-009"` as protocol.md §Initialization step 8 instructs. The harness ignored the requested name and assigned a random slug `delegated-frolicking-stonebraker`. All downstream references — `~/.claude/teams/<name>/config.json` lookups, Agent spawn `team_name` parameter, inbox discovery — had to use the harness-assigned slug, not `req-009`.
- **Classification**: systemic (documentation correctness bug)
- **Root cause**: protocol.md §Initialization step 8 and the guidance in §"If team state appears inconsistent or spawn errors" and §"How to check idle agent state" all assume the caller controls the team name. In practice, `TeamCreate`'s `team_name` parameter is advisory — the harness assigns its own slug. The downstream advice "inspect `~/.claude/teams/{req-id}/config.json`" will resolve to a non-existent path for anyone who reads the doc literally.
- **Root cause file**: `${CLAUDE_PLUGIN_ROOT}/references/team-mode/protocol.md`

### Friction 2 — SKILL-body.md "That is not a reminder; it is the enforcement" is not true when invoked from main Claude
- **What happened**: During Initialization, Team Lead needed to list `.couch/requirements/` to pick the next `req-NNN`. Glob with pattern `.couch/requirements/req-*` returned "No files found" (likely gitignore interaction). Team Lead fell back to `Bash: ls .couch/requirements/`, which succeeded — because in this session the skill was invoked from the main Claude context where the frontmatter `disallowedTools` and Write hook were not enforced at runtime.
- **Classification**: systemic (documentation correctness bug + minor protocol gap)
- **Root cause**: Two gaps. (a) SKILL-body.md §"You don't write code" overstates enforcement with the sentence "That is not a reminder; it is the enforcement." When the skill is loaded via the plugin, hooks enforce; when the skill-body guidance is being followed from main Claude, the harness lets disabled calls through. The absolute claim erodes trust in the doc. (b) protocol.md §Initialization step 3 tells Team Lead to "check existing `.couch/requirements/` dirs" without specifying a Glob-safe pattern. Since `.couch/requirements/` is gitignored in some setups and Glob's default `**/*` semantics don't reliably match directories, Team Lead has no clean tool-limited path and will reach for Bash.
- **Root cause file**: `${CLAUDE_PLUGIN_ROOT}/references/team-mode/SKILL-body.md` and `${CLAUDE_PLUGIN_ROOT}/references/team-mode/protocol.md`

### Friction 3 — TaskCreate assigns ids in reverse creation order (NIT)
- **What happened**: Team Lead created Task A (Coder) first, then Task B (Architect). The harness assigned Task A → id `2` and Task B → id `1`. Minor confusion when setting up blockedBy relationships.
- **Classification**: exploratory / harness behavior
- **Root cause**: Harness-side TaskCreate id assignment. Not an agent-framework concern.
- **Root cause file**: N/A (upstream Claude Code harness)

## Improvement Proposals

### Proposal A — Document that TeamCreate `team_name` is advisory
- **Target file**: `${CLAUDE_PLUGIN_ROOT}/references/team-mode/protocol.md`
- **Section**: §Initialization, step 8 (and downstream references in §"If team state appears inconsistent or spawn errors", §"How to check idle agent state", §Spawn Prompt Template)
- **Current text** (step 8):
  > `TeamCreate` with `team_name: <req-id>` — this creates the shared task list and messaging channel for the swarm
- **Proposed text** (step 8):
  > `TeamCreate` with `team_name: <req-id>` as a hint — the Claude Code harness may ignore the requested name and assign its own random slug (e.g., `delegated-frolicking-stonebraker`). Capture the slug the harness actually returns and use it as `<team-slug>` for every downstream reference: all `Agent` spawn `team_name:` parameters, `~/.claude/teams/<team-slug>/config.json` path discovery, and inbox lookups. Record the mapping `<req-id> → <team-slug>` in `run.json` under a new `team_slug` field so later agents and researchers can find the team state.
- **Also update** the guidance in §"If team state appears inconsistent or spawn errors" item 2 and §"How to check idle agent state" item (1): replace every literal `~/.claude/teams/{req-id}/config.json` with `~/.claude/teams/<team-slug>/config.json` and add a one-line note: "Note: `<team-slug>` is the harness-assigned name captured at TeamCreate time, which may differ from `<req-id>`."
- **Rationale**: The current text guarantees that anyone who follows protocol.md literally will fail to find team state, and any research brief or upstream bug report that cites `team req-NNN` will be wrong. This is a pure documentation correctness bug with zero ambiguity.

### Proposal B — Soften the "That is not a reminder; it is the enforcement" claim in SKILL-body.md
- **Target file**: `${CLAUDE_PLUGIN_ROOT}/references/team-mode/SKILL-body.md`
- **Section**: §"You don't write code"
- **Current text**:
  > The frontmatter of this skill disables the tools that would let you touch project state directly — `Edit`, `Bash`, a Write hook restricts `Write` to orchestration paths only. That is not a reminder; it is the enforcement. If you find yourself reaching for a tool and it isn't there, that is the system telling you the work belongs to an agent.
- **Proposed text**:
  > The frontmatter of this skill declares the tools that would let you touch project state directly — `Edit`, `Bash`, Glob, Grep — as disallowed, and a Write hook restricts `Write` to orchestration paths only. These restrictions are enforced by the plugin hooks when the skill is loaded through the plugin loader. If you are operating from main Claude, the harness may not block the calls at runtime — treat the frontmatter as a rule you must follow anyway: if you find yourself reaching for `Edit`, `Bash`, `Glob`, or `Grep`, that is the system telling you the work belongs to an agent, even if the harness lets the call through. Spawn or message one. Read is path-restricted by the harness, so the files you can see are the orchestration state files you need to run the workflow.
- **Rationale**: The absolute "it is the enforcement" sentence is factually false in the main-Claude invocation path observed this run. Softening it to "rule you must follow anyway" preserves the intent (tool-boundary discipline) without making a claim the harness doesn't back up.

### Proposal C — Give Team Lead a tool-limited way to list `.couch/requirements/` directories
- **Target file**: `${CLAUDE_PLUGIN_ROOT}/references/team-mode/protocol.md`
- **Section**: §Initialization, step 3
- **Current text**:
  > Generate `req-<NNN>` (check existing `.couch/requirements/` dirs)
- **Proposed text**:
  > Generate `req-<NNN>` by listing existing requirement dirs with `Glob` using the pattern `.couch/requirements/req-*/` (trailing slash matches directories and bypasses gitignore interactions with file-level globs). Take the highest `NNN` and increment. Do NOT fall back to `Bash: ls` — if `Glob` returns no results, treat that as "no requirements yet" and start from `req-001`.
- **Rationale**: Eliminates the Bash-fallback temptation by giving Team Lead an explicit, tool-boundary-respecting pattern. Paired with Proposal B, which softens enforcement language, this gives Team Lead a clean path to comply.
