# Retrospective: req-016

## Summary

req-016 landed the fact-verification rule for adversarial review: a mandatory requirement that architects and challengers verify factual claims about codebase state, harness behavior, protocol text, prior retro content, or session-evidence generalizations before accepting or rejecting them during debate. Three files were edited: `agents/architect.md`, `references/team-mode/souls/challenger.md`, `references/multi-agent-mode/souls/challenger.md`. The requirement was planned by architect-3, reviewed by a challenger who applied the new rule in vivo and caught two imprecisions, and finalized by plan-reviser (a restart agent) after the original team was lost to a harness namespace collision bug. The rule was then verified at the review gate by plan-reviser reading each file directly — the rule dogfooded itself at every stage.

## Friction Analysis

### Friction 1 — Cross-project team namespace collision (systemic / upstream harness bug)

- **What happened**: The Claude Code harness has a global `~/.claude/teams/` namespace. `TeamCreate` with `team_name: "req-016"` collided with another project's req-016 team from two days prior. The harness renamed the new team to `misty-napping-gadget`. However, subsequent `Agent` spawn calls using `team_name: "req-016"` did NOT honor the rename — they silently joined the OTHER project's `req-016` team instead. Architect-3 was spawned into the wrong project's team.
- **Evidence**: `~/.claude/teams/req-016/config.json` contained architect-3 with `cwd: /Users/glitchboyl/work/skills/couch-potato` alongside agents from `/Users/glitchboyl/work/deevid-frontend-v2`. The message Team Lead sent to architect-3 landed in `misty-napping-gadget/inboxes/architect-3.json`, but architect-3 was listening in `req-016/inboxes/` — unidirectional message loss. The debate was completed by architect-3 and the challenger, but the post-review routing was lost entirely.
- **Classification**: systemic / upstream harness bug
- **Impact**: Lost one full architect-challenger debate cycle. The restart (this run) required re-spawning a new plan-reviser agent in a new project-qualified team (`couch-potato-req-016`). Artifacts survived because they were written to the filesystem, not held only in agent memory.
- **Mitigation used**: Project-qualified team name `couch-potato-req-016` for the restart, which avoided the collision.
- **Proposal (follow-up requirement, NOT landed here)**: couch-potato's protocol should default to project-qualified team names to prevent cross-project namespace collisions. Format TBD — this requires its own requirement and scoping.

### Friction 2 — Sycophancy caught empirically, not hypothetically

- **What happened**: The req-015 retro named a three-LLM sycophancy chain as a process failure and proposed the fact-verification rule. In this very session, Team Lead attempted to disprove Challenger's session-evidence claim with the recall "I ran TeamCreate four times and none were overridden" — a generalization from n=4. req-016's own TeamCreate call then hit the collision-rename, proving the random-slug behavior exists in the current harness. The session-evidence-recall gap (Revision 1) is the rule closing the exact loophole through which this mistake entered.
- **Classification**: systemic (adversarial review process gap, same root cause as req-015 Friction 2)
- **Impact**: No product impact. The empirical contradiction strengthened the requirement's rationale and directly validated Revision 1 (session-evidence generalization as a factual claim requiring sample-size verification).
- **Note**: This is the first recorded instance where a rule-in-progress was empirically validated by a failure that occurred during the same session that was designing the rule. The rule is not hypothetical.

### Friction 3 — Debate routing fragility

- **What happened**: Architect-3 and the challenger completed a full debate, reached consensus on three revisions, and wrote the verdict to `challenger-verdict.md`. But the message routing from challenger back to team-lead was broken by the namespace collision, so Team Lead never received confirmation and could not proceed. The entire debate cycle was lost at the routing layer, not the content layer.
- **Classification**: structural vulnerability (multi-turn debate dependent on message delivery)
- **Impact**: One full debate cycle wasted. Recovery was possible only because the challenger wrote `challenger-verdict.md` to the filesystem as an artifact before the routing failure.
- **Proposal (follow-up consideration, NOT landed here)**: Any multi-turn debate whose resolution depends solely on SendMessage delivery is vulnerable to routing failures. Consider periodic artifact checkpoints — write current debate state to a file at each round — so that any restart agent can pick up where the debate left off without re-litigating. The challenger's pre-emptive `challenger-verdict.md` write was the saving grace here; making this a protocol expectation rather than an accident would eliminate this failure mode.

## Positive Notes

### Rule validated in vivo at the review gate

This is the first requirement in the self-host arc where the fact-verification rule caused the adversarial review to catch real imprecisions in architect-3's planning artifacts. Challenger read the cited files directly and found two items: (1) the "included verbatim" citation pointed to the wrong line (permanent vs. temporary agent spawn), and (2) the "never landed" framing of req-009 Proposal A was wrong — it had partially landed in SKILL-body.md via req-015 task-002. Both catches were minor but would have been invisible in a non-verifying review. The rule works.

The rule also validated itself at this review gate: plan-reviser was required (by the challenger-verdict instructions) to verify challenger's own claims before applying them. Every claim checked out. This is the dogfood chain working correctly: the new rule, applied to the review of the new rule, caught no further errors — which is itself a data point.

## Convergence Test (req-012)

req-016 passes. The fact-verification rule is novel in the mandatory read path (`agents/architect.md` is a mandatory agent definition; both challenger SOUL files are included in temporary discussion agent spawn prompts per protocol). Nothing redundant was added.

Calibration note carried forward from req-015 Friction 3: the convergence test is a redundancy filter, not a correctness filter. The sycophancy event in req-015 showed that a rule present in the mandatory path can still be wrong-by-omission (sample-size gap was not covered until Revision 1). The test is necessary but not sufficient for correctness. This observation does not require a standalone proposal — it is a standing calibration note on the test's scope.
