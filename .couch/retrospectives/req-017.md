**Classification**: planning over-engineering (architect-initiated, challenger-uncontested)

The req-015 "team_name MUST equal req-id" rule was itself unnecessary. This means a prior mandatory-path rule was not just wrong-by-omission (req-015 Friction 2) but wrong by over-specification.

**Classification**: prior-req latent scope error, surfaced and corrected here

The final plan consumed 2 tasks, 2 file edits, 0 debate rounds. The scope reduction was not discovered through process — it required user intervention.

## Meta Finding — Standing Diagnostic gap

The convergence test (req-012) asks: "is this rule already in the mandatory path?" The fact-verification rule (req-016) asks: "is this factual claim verified by direct read?" Neither asks: "does this proposed rule prevent a concrete failure mode that existing rules do not?"

**Proposed third diagnostic question** for future Architect planning and adversarial review:

> For each proposed rule or new field, name the concrete failure mode it prevents. If the failure mode is already prevented by an existing rule or by harness behavior, the proposed rule is scope creep.

This question would have caught: (a) the req-015 "MUST equal req-id" over-specification — the harness's capture-the-return rule already handles the failure mode; (b) the req-017 v1 `project_slug` machinery — the harness auto-rename already handles cross-project collision uniqueness.

Fact-verification is orthogonal to scope discipline. Verifying each other's facts does not prevent two architects from converging on over-engineering that neither questions.
