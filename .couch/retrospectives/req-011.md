# Retrospective: req-011

## Summary

req-011 closed two documentation gaps in `references/team-mode/SKILL-body.md` (Proposals A and B from the req-010 retrospective) and fixed the global-PreToolUse hook bug that had been blocking all agents — including Architect reads — since the hooks were first registered. This is the first requirement in the self-host session where the swarm actually wrote production code and the changes landed correctly.

## Friction Analysis

### Friction 1 — First Architect spawn ran foreground and never joined the team
- **What happened**: The initial Architect spawn for req-011 planning ran as a foreground agent in the Team Lead's session rather than as a proper `TeamCreate`-joined teammate. The user caught this.
- **Classification**: execution error (user-caught)
- **Root cause**: Team Lead spawned Architect before confirming team initialization was complete. This is exactly the Friction 1 pattern from req-010 — the Proposal A gate was not yet in place when this run started (it was being written by this very run). The irony is noted.
- **Impact**: Low — Architect still produced correct planning artifacts. The peer channel was unavailable but Architect's role in planning (produce requirement.md + tasks.json) doesn't require inter-agent messaging. User caught it before Execute began.

### Friction 2 — Architect initially proposed `author_mode_paths` as the hook-wall fix (wrong root cause)
- **What happened**: During planning, Architect framed the hook wall as a self-hosting scope gap and proposed Proposal C (`self_host` / `author_mode_paths` config flag) as the fix. The user diagnosed the real root cause: the hooks were registered as global `PreToolUse` with no skill-context matcher, making them inherently overreaching. The correct fix was deletion, not configuration.
- **Classification**: planning error (user-caught)
- **Root cause**: Architect reasoned from the observed symptom (hooks block self-host file writes) rather than from the mechanism (global hook scope). The req-010 retro described the hooks as "doing exactly what they should for normal projects" — a framing that anchored toward configuration rather than deletion. The user's diagnosis broke that anchor.
- **Impact**: Medium — required a scope revision mid-planning (task-002 changed from "add author_mode_paths feature" to "delete the global registration"). User caught it before any Coder was spawned.
- **Lesson**: When the symptom is "hook blocks files it shouldn't," the first question should be "why is this hook registered globally?" not "what whitelist entry is missing?"

### Friction 3 — Bootstrap required manual Team Lead intervention with explicit user authorization
- **What happened**: Both target files (`references/team-mode/SKILL-body.md` and `hooks/hooks.json`) were outside `.couch/` and `.claude/`, so the very bug being fixed blocked Coders from reading and writing the fix. A one-shot manual bootstrap was required: user edited `hooks/hooks.json` by hand to remove the `PreToolUse` block, then ran `/reload-plugins`. After that, Coders ran normally.
- **Classification**: structural (chicken-and-egg bootstrap — inherent to self-host requirements that touch the hook system)
- **Root cause**: There is no mechanism for a Coder to modify hook registration while that registration is active and blocking writes. Any requirement that fixes a global hook wall will require a user-authorized bootstrap step to break the cycle.
- **This is a special case of Proposal B**: the user performed the "apply manually outside the swarm" option — but with explicit authorization documented in `requirement.md` as a planned prerequisite, not as a silent violation. This is the correct form of that option and should be the template for future self-host hook-system requirements.
- **Impact**: Low — the bootstrap was documented in `requirement.md` before Execute began, the user confirmed completion before Coders were spawned, and task-002 verified the state cleanly. No silent violations.

## Improvement Proposals

### Proposal A — Document the "authorized bootstrap" pattern for hook-system requirements
- **Target file**: `${CLAUDE_PLUGIN_ROOT}/references/team-mode/SKILL-body.md` (or `workflow.md`)
- **Proposed addition**: A note in the Situations section or workflow docs covering "requirement targets the hook system itself (chicken-and-egg bootstrap)." The documented pattern: (1) Architect names the bootstrap in `requirement.md` as an explicit prerequisite; (2) Team Lead confirms with the user before Execute; (3) user performs the bootstrap and confirms; (4) Coders run normally; (5) one task verifies the bootstrap state is clean. This is a user-authorized override, not a Proposal B violation.
- **Rationale**: req-011 established this pattern correctly. Documenting it prevents future runs from treating the bootstrap as either a violation (applying it silently) or a blocker (refusing to proceed at all).
